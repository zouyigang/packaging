# 装箱算法性能优化

记录求解器的性能设计与实测结论。修改 GA、极点放置、几何检测、评估器或前端求解体验时同步更新。

原则：**所有提速必须是等价改写**——布局签名、装载率、约束行为不得改变。
凡是会改变解的优化，走 `docs/industrial-strategies.md` 的质量门禁，而不是当成性能优化。

## 可观测性

求解响应返回 `performance.runtime_ms` / `stages_ms` / `counters`，覆盖构造 placeables、
单容器装载、`find_placement`、evaluator、GA 每代。`scripts/benchmark_solver.py` 做固定样例的
重复性能采集。前端结果面板有性能诊断区（默认折叠）与慢求解提示。

## 已落地的优化

**`find_placement`（所有策略共用的热点，占大头）**

- 预计算每个货品的可用朝向尺寸；极点去重（内部 key set，O(1)）、越界裁剪、尺寸感知过滤。
- 放置后删除最小角已被新箱体覆盖的极点。
- 重叠检测：z 区间层（按 bottom z 排序，不可能相交时提前停止）+ x/y 网格索引；
  候选查询只扫相交网格。支撑检测同样走 z 顶面 + x/y 网格。
- 评分下界提前停止：候选按目标评分稳定排序，找到可行最优后，评分不可能更优的候选
  不再做碰撞/支撑/重心校验。
- 空间利用类默认评分 `(z, y, x)` 走点序扫描快路径，避免对「点 × 姿态」全量建堆排序。
- 候选尝试用轻量 tuple 而非 dataclass；高频 counters 本地累加后批量写回。
- 硬约束合并为一次 supporters 循环，地面放置 / 不可堆叠 / 缺失支撑层早返回。

**GA**

- 档位 `ga_speed`（快速 16/8-12、标准 32/20-30、精细 64/40-60）+ 无提升早停。
- 排序签名缓存：相同放置顺序的个体不重复 decode。
- population 多进程并行，worker 数 `min(cpu_count - 1, 4)`，小 population 自动回退单进程。

**其他**

- 装载顺序重排 `_resequence_inside_to_outside` 改用 z 顶面 + x/y 网格查询。
- 前端：3D 场景缓存箱体位置/尺寸/颜色；结果面板与 2D 俯视缓存派生数据；Vite 手动分包
  （主业务包约 52KB，无大 chunk warning）。

## 试过但不保留（别再走一遍）

- **`stability` 按 z 层 + 底面积顺序扫描的快路径**：实测硬检查次数上升、总耗时反而变慢。
- **按剩余货品尺寸安全裁剪极点**：维护 suffix 尺寸集合的开销超过收益。
- **只在重心策略传入 `balance_points`**：其他策略也有重心硬约束，直接移除 fallback 会
  破坏既有约束测试。要做得先拆分「硬重心约束」与「重心目标额外候选」的职责。
- **GPU**：装箱逻辑是 Python 分支 + 对象操作 + 极点遍历 + 碰撞检测，不是矩阵计算型任务。
  引入 GPU 不会自动提速，得先把几何检测和候选评分重写成批量数组/CUDA，投入大、收益不确定。
  只有当核心逻辑可批量化之后才值得重新评估。

## 2026-07-14 顺序配送评分与碰撞热点优化

顺序配送消除多用容器后，4 个托盘块散成约 184 个独立单元，1140 件耗时从 7.90s 涨到 11.74s。
本轮压回 8.66s，且**四个策略的布局签名逐字节不变**。

热点定位（cProfile，1140 件顺序配送）：`find_placement` 占 10.2s/11.4s，其中
`overlaps_existing` 5.7s、评分链路 9.8s（累计）。四处改动：

1. **配送簇质心按世代缓存**（`ScoreContext.cluster_targets` + `epoch`）。质心在一整次
   `find_placement` 里恒定，原实现却对每个候选箱重建组键、查字典、重算均值——259 万次。
2. **评分器按入口配置特化**（`LoadingEfficiency.make_scorer`）。入口面分派原先每个候选箱
   都跑一遍字符串比较链；居中分母、站点跨度同理。全部提到构造期。顺带发现单入口时
   「该面深度」与「最近入口深度」是同一个值，原先各算了一遍。
3. **重叠检测去掉跨分箱去重**。实测去重集几乎去不掉任何重复（723 万次 `set.add` 对
   682 万个待扫物体），而重叠检测一发现重叠就返回，重复看到同一个物体不影响结果。
   支撑扫描**保留**去重：那里重复会把支撑面积算两遍。
4. **循环开销**。`overlaps_existing` 原先每个 z 层都重建两个 `range`，而绝大多数箱子只落在
   单个分箱里；`build_attempts` 每个候选箱一次函数调用。均已展开。

| 1140 件 | 优化前 | 优化后 |
| --- | ---: | ---: |
| `delivery_sequence` | 11.74 s | **8.66 s** |
| `cost_efficiency` | 7.16 s | 6.35 s |
| `space_utilization` | 7.16 s | 6.36 s |
| `safe_loading` | 5.05 s | 4.50 s |
| 48 件配送门禁 | 51.0 ms | 39.9 ms |

第 3、4 项在所有策略共用的热路径上，故非配送策略同步受益。

## 当前耗时

1140 件默认样例（开发机观测值，仅供同环境对比）：

| 策略 | 耗时 |
| --- | ---: |
| `cost_efficiency` / `space_utilization` | 约 10.3 s（含多容器类型试装，见下） |
| `safe_loading` | 约 4.5 s |
| `delivery_sequence` | 约 8.7 s |
| `ga_fast` | 约 1.1 s |

成本/空间策略耗时高于其他策略，是**多容器类型选择的固有开销**：每次开箱要对每个候选
类型各试装一遍。默认样例有 20GP / 40GP 两种类型，所以试装量翻倍。这笔钱换来的是
容器数 2→1、成本 4000→3400，见 `docs/industrial-strategies.md`。

耗时不纳入质量门禁——它随机器波动，误报会让人开始忽略门禁。
