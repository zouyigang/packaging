# 工业装载策略与校验

## 策略目录

生产界面使用四个策略：

- `cost_efficiency`：先满足必装、优先级和装载完成度，再最小化容器启用成本与托盘处理成本。
- `space_utilization`：对可用容器做试装，选择预计填充率最高的最佳适配容器。
- `safe_loading`：同时优化局部堆垛稳定与整体载荷分布，先保护两项中的较弱项。
- `delivery_sequence`：按 `stop_seq` 升序卸货，禁止倒货，检查卸货路径和支撑依赖。

`custom` 是高级策略；完成度和工业安全硬约束不参与权重交易。托盘化通过
`pallet_policy=auto|prefer|avoid|required` 单独配置，不再使用“托盘越多越好”的评分。

所有策略共用以下字典序：硬约束、必装货品、业务优先级、装载件数/体积、策略目标、确定性并列规则。

## 标准与工业模式

`validation_mode=standard` 保持宽松兼容：缺失成本时使用容器数代理并给出告警。

`validation_mode=industrial` 要求：

- 货品提供实际重量；
- 通用设备提供重心范围；
- 道路车辆提供纵向载荷分布曲线；
- 设备提供地板载荷、运输加速度和摩擦系数；
- 成本策略提供容器启用成本和可用托盘处理成本。

ISO 集装箱缺少重心范围时会应用可编辑的模板值，并明确提示需要承运方确认。结果只表示对已配置规则的计算校验，不构成法规认证或绑扎施工设计。

## 工业指标

每个已用容器返回：

- 三轴归一化重心；
- 纵向载荷分布曲线裕量；
- 最大地板载荷；
- 加速度/摩擦条件下的最小方向固定能力；
- 简化抗倾覆裕量。

承重沿直接支撑关系递归传递到底层。顺序配送还会在每个站点卸货后重新检查剩余载荷的重心、载荷曲线和地板载荷。

## 混合组托

相同 `stop_seq` 且具有相同非空 `pallet_group` 的货品可以混托；空兼容组保持同品组托。混托仍遵守托盘限重、限高、朝向、支撑和顶部承重约束。托盘块在容器内只允许水平 90° 旋转。

## 兼容名称

后端继续接受旧目标，并在评估中返回 `objective_requested`、`objective_resolved` 和 `strategy_profile`：

- `transport_cost`、`min_containers` → `cost_efficiency`
- `max_utilization` → `space_utilization`
- `load_stability`、`stability`、`weight_balance`、`center_of_gravity` → `safe_loading` 的兼容预设
- `loading_efficiency`、`multi_customer_delivery` → `delivery_sequence`
- `advanced_score`、`balanced` → `custom`

## 每轮优化的交付与汇报格式

每一步优化必须包含代码、测试、文档和验证结果；不以“代码已修改”作为完成标准。
每轮结束时固定输出以下三部分：

### 本轮做了什么

- 列出实际完成的算法、模型、接口、界面或文档改动；
- 列出新增或修改的关键文件；
- 列出已运行的测试、构建、基准和浏览器验证；
- 记录关键指标变化，以及已知但不阻断交付的问题。

### 还剩什么

- 列出当前阶段尚未达到完成标准的事项；
- 区分功能缺口、测试缺口、性能风险和数据配置缺口；
- 明确任何依赖用户输入、工业参数或外部标准确认的事项。

### 下一步做什么

- 只指定一个最高优先级的下一项工作；
- 写明目标、主要改动范围和完成标准；
- 如果下一步存在前置条件，先说明需要满足的条件。

除非发现阻断性设计问题，实施顺序按第 0 步到第 6 步推进。每完成一步，更新本节状态并记录测试与基准结果。

## 基准样例与门禁

两个基准样例，均在 `scripts/benchmark_solver.py`；当前指标基线钉在该脚本的
`QUALITY_BASELINES` 里，是唯一的事实来源（本文档不再另抄一份，避免两处漂移）。

- **48 件快速门禁**（`--industrial-strategies`）：两个卸货站点、必装高优先级货物、
  实际重量、设备重心范围、地板载荷、运输加速度、摩擦系数、启用成本、后门入口。
- **1140 件工业验收**（`--industrial-large`）：前端默认样例（40/300/300/500 四类货物），
  两种设备类型（20GP / 40GP）。含 `safe_loading` 的安全优先路径，共 9 个用例。

门禁强制：全部装完、`feasible`、无工业错误码、无托盘悬空、容器数不超过 3、
两次运行布局签名一致，以及**关键指标不得超出允许退化幅度**（见「策略质量退化门禁」）。

## 增量工业载荷上下文

`IndustrialLoadContext` 是构造阶段与最终校验**共享的同一个计算核心**——这是刻意的，
避免候选阶段与结果阶段的公式漂移。

- `preview` 预测候选放置后的指标且不改状态，`commit` 确认后增量更新质量、三轴质量矩、
  最小摩擦系数；托盘块用 `preview_batch`（托盘底座 + 自重 + 全部内部货物）。
- 载荷沿直接支撑关系递归分摊到底层，据此算最大地板载荷；按纵向重心插值设备载荷曲线。
- 标准模式不创建上下文，也不付这份开销。构造阶段指标以 `construction_` 前缀存入容器
  工业指标，便于与最终复核结果对照。
- **托盘载荷口径**：托盘自重作用于台面高度中点，连同货物载荷通过**完整托盘底面积**
  传递到地板；内部货物仍按各自坐标/重量/摩擦系数计算，不用货物包围盒代替托盘底面。

工业模式下已前置为硬约束的：设备三轴重心范围、最大地板载荷、道路车辆纵向载荷曲线。
候选超限时继续试同一容器的其他位置和方向，本容器无解则转下一容器；所有容器都无解时
返回 `COG_OUT_OF_RANGE` / `FLOOR_LOAD_EXCEEDED` / `LOAD_DISTRIBUTION_EXCEEDED` 具体
错误码，而不是无原因的 `partial`。

## 托盘承载边界与可视化一致性

工业默认规则禁止货物底面超出托盘有效承载边界。最终校验逐件比较货物定向尺寸与明确的
托盘实例边界；发生越界时返回 `PALLET_OVERHANG` 并将方案标记为 `infeasible`。48件和
1140件策略基准均输出 `pallet_overhang_count`，大规模验收要求该值为零。

前端 3D 和 2D 视图必须使用后端返回的 `pallet_instances` 绘制托盘原点、方向和实际底面
尺寸。禁止再根据当前回放中可见货物的最小坐标反推托盘位置，因为回放早期只显示部分货物，
且 `WLH` 旋转会交换托盘长宽，反推会产生货物大面积悬空的假象。旧版响应没有实例数据时
保留兼容回退，但只用于历史结果展示。

使用前端默认1140件、顺序配送、标准校验和托盘自动配置复核：全部1140件装入3个容器，
4个托盘实例均无真实悬挑，`pallet_overhang_count = 0`。修复后的3D回放会在只显示部分托盘
货物时仍保持托盘真实原点和旋转后尺寸。

## 重心可达区间与两阶段搜索

成本、空间和安全策略采用两阶段重心候选搜索：

1. 第一遍要求放置后的即时重心处于设备允许范围，优先保持全过程载荷平衡；
2. 第一遍确实因重心约束无解时，才计算剩余货物质量和各方向最小尺寸形成的最终重心可达区间；
3. 可达区间与设备范围有交集时，允许临时偏载候选；否则继续拒绝；
4. 最后一件没有剩余修正质量，因此仍执行严格最终重心约束。

可达区间使用保守几何边界，不假定具体后续位置。回归样例覆盖高而轻的首件临时造成高
重心、后续低而重货物能够把最终重心降回限制内的场景，并确认恢复遍不会污染首次搜索状态。

恢复遍只在本件首次搜索实际出现 `COG_OUT_OF_RANGE` 时触发；不能因为容器空间已满、碰撞、
支撑或入口问题重复扫描。该触发门禁将1140件成本/空间策略从错误实现的约10.8秒恢复到
约7.1秒，安全策略恢复到约5.0秒，布局签名和最终质量指标不变。

顺序配送不使用上述两阶段搜索，而是使用逐站点载荷上下文统一判定重心（见下节）：
直接用全体剩余货物做恢复判断会掩盖某一站卸货后的剩余载荷偏心，已被回归证明可能
产生 `POST_DROP_COG_OUT_OF_RANGE`，因此顺序配送按「卸完每一站之后的剩余载荷状态」
分别维护质量矩与可达区间。

## 顺序配送逐站点载荷上下文与通道前置

`_DeliveryStopCogTracker` 为工业模式顺序配送维护一组载荷状态：t=0 为全载，
t=站点号 为卸完该站及更早站点后的剩余载荷；最后一站卸完为空载不检查。每个候选放置
必须让所有受影响状态（候选货品仍在船上的状态）满足：

1. 状态重心处于设备允许范围内；或
2. 可由「尚未放置且比该站更晚卸货」的货物按保守几何界修正回范围内。

修正池按站点过滤并随放置进度递减；某状态的修正池耗尽时自动收紧为严格检查，
因此最终布局与逐站卸货后的剩余载荷都满足设备重心范围。全载状态同样由该跟踪器
以「范围内或可修正」判定，取代原先逐候选的严格全载检查——顺序配送按站点聚集
的构造过程天然存在中途偏心，严格检查会大量拒绝本可修正的高分候选。

放开重心后，直线卸货通道检查同步前置到构造过程（与最终 `DELIVERY_PATH_BLOCKED`
校验同口径）：候选必须存在一个对齐入口、通道上没有更晚卸货的已放置货物（同站
货物可先行移除，不算永久阻挡），且不能堵死任何已放置的更早卸货货物的最后一条
畅通出口。求解器为每个已放置单元维护「仍畅通出口集合」，随放置增量收缩。标准
模式保持事后诊断，不启用该候选过滤。

新增计数器：`industrial_preview_post_drop_cog_recoverable`、
`industrial_preview_post_drop_cog_exceeded`、`delivery_corridor_rejections`；
无解时的明确错误码新增 `POST_DROP_COG_OUT_OF_RANGE` 与 `DELIVERY_PATH_BLOCKED`。

## 顺序配送多用容器的定位与消除

顺序配送在 1140 件基准上长期比成本策略多用 1 个容器（3 vs 2）。定位过程推翻了两版
错误归因，这里记录被实测排除的假设，以免重走：

- **不是重心拒绝**：标准模式顺序配送、去掉 `cog_limits` 的工业顺序配送均为 3 个容器。
- **不是地板不够**：三只容器地板占用实测为 81% / 81% / 13%，两只主容器各有约 2.7 m²
  空地。据此实现的「按站点地板/深度预留」产出了逐字节相同的布局（纯粹的空操作），
  已整体回退。缺的从来不是地板预算，是地板**利用率**。

真实成因是两个各自独立、都在浪费柱高与地板的构造缺陷：

1. **横向居中把宽度压成废条**。端门（`x_min`/`x_max`）容器内全部货物同站时没有卸货
   顺序可优化，而 `LoadingEfficiency` 的评分键仍以「距宽度中线的距离」为主项，于是
   货物堆向宽度中线，两侧各留下放不下一列货的窄条。改为贴墙紧凑键
   `(z, y, -depth, cluster, x, -area)` 后同宽度能多排一列。顶开（`z_max`）与侧门
   （`y_min`/`y_max`）的居中是有意的取货可达性设计，保持不变（见 `test_packer` 用例）。
2. **`auto` 码托盘选了会废掉柱子的封顶托盘块**。托盘块是 `max_load_top=0` 的封顶块，
   谁也压不上去，所以它脚下那根柱子从 `max_stack_height` 到容器顶的余高是永久损失。
   原判据 `load_efficiency >= 0.45` 只衡量托盘包围盒**内部**填得满不满，看不见外部
   废掉的柱高：默认样例的托盘块净码高 1500mm（台面 150 + 限高 1500，块高 1650），
   而块内的 box-B 直接堆进 2390mm 高的容器能摞 7 层 = 2100mm——`load_efficiency`
   高达 0.84 的「优质」托盘块，实际把 1.2 m² 地板上方 29% 的柱高扔掉了，4 块即
   4.8 m² 地板集体降柱。

对应引入 `column_efficiency = 托盘净码高 / 同货直接堆叠可达净高`（`packer.
_pallet_column_efficiency`），作为第三个参数传入 `Objective.should_palletize`。
每层水平填充率在两种方案里相同，在比值中约掉，故该判据只需高度信息。装卸效率类
目标（含顺序配送）要求 `column_efficiency >= 0.85`——愿意为「一次叉走一整块」付出
一点柱高，代价过大时宁可散装堆到容器顶；`stability` / `safe_loading` 要的是「整块
不散」，甘愿付出柱高，故不看该判据；`pallet_policy` 为 `prefer` / `required` 时仍
无条件尊重用户意愿。

1140 件工业验收基准（双次运行签名一致）：

| 指标 | 改造前 | 改造后 |
| --- | ---: | ---: |
| 容器数 | 3 | **2** |
| 总成本 | 6080 | **4000** |
| 体积利用 | 34.64% | **51.97%** |
| 装卸匹配 | 56.61% | 55.57% |
| 稳定性 | 38.28% | 37.26% |
| 平均耗时 | 7.90 s | 11.74 s → **8.66 s** |

## 安全装载的密度/安全交换与 `safety_priority`

承接上一节的否定结果：所有能改善安全指标的杠杆——同档位置细化、固定力边际细化、
放置顺序——**无一例外都要多花一个容器**。扫过的排序里，只有那些碰巧复现出原排列的
（按体积、按底面积、按重量——这批货恰好同序）才保持 2 箱；任何真正改变顺序的排序
都至少 +1 箱。

结论不是「算法还没调好」，而是**这批货的密度与安全本身互斥**：40 件封顶大箱吃掉大半
地板，其余货只能往高里码；而 box-D 底面仅 200mm，码高必然细长。要堆垛不细长，就得
少码高、多占地板——那就是多一个容器。

因此这是产品决策而非算法问题，做成开关 `SolveRequest.safety_priority`（默认 `false`），
只影响 `safe_loading`：

| 1140 件 | 容器 | 成本 | 所需固定力 | 危险簇 | 细长比 | 稳定性 | 耗时 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| `cost_efficiency`（对照） | 2 | 4000 | 14.46 kN | 26 | 10.5 | 0.432 | 6.35 s |
| `safe_loading`（默认，容器优先） | 2 | 4080 | 11.67 kN | 27 | 11.0 | 0.426 | 4.51 s |
| `safe_loading` + `safety_priority` | 3 | 6080 | **5.53 kN** | **17** | **8.0** | **0.440** | 2.45 s |

默认路径已经比成本策略少 19% 的所需固定力（11.67 vs 14.46 kN），只是危险簇数量略高；
开启安全优先后固定力再降 53%（相对默认）/ 62%（相对成本策略），代价是 +1 个容器、
+2000 成本。想要密度的用户走 `cost_efficiency` / `space_utilization`（仍 2 箱）。

实现是 `SafeLoading.order_placeables` 的一条替代排序：按**扁平度**
`最小边 / √底面积` 升序——矮胖件先铺开、细高件留到最后。堆垛簇的倾覆裕量与所需固定力
由「重心高 / 底面半宽」决定，先落位的扁平件把底面拓宽、把重心压低。
`get_objective(..., safety_priority=True)` 另建实例，不污染注册表单例；`pallet_policy`
为 `prefer` / `required` 等其他开关互不影响。

1140 件工业验收基准新增 `industrial_large_safe_loading_safety_first`，与其余四例同样
校验装载完成、无工业错误码、布局签名确定性与容器上限，两条路径都在门禁内。

## 安全装载：倾覆裕量为什么不能转正（否定结果）

第 3 步原定目标是「让最弱安全项（倾覆裕量）转正」。**该目标对本样例在物理上不可达**，
这里记录推翻过程，避免重走。

先摆事实：`safe_loading` 目前在它本该优化的指标上**并不比 `cost_efficiency` 好**——
1140 件基准上危险堆垛簇 27 vs 26、细长比 11.0 vs 10.5、簇倾覆裕量 -7.8 vs -7.4。
根因是 `SafeLoading.make_scorer` 直接返回 `placement_score = (z, -底面积, y, x)`，
一个纯几何键，对任何安全指标都毫无感知。这个缺口是真的。

但「让裕量转正」不是它的解。倾覆裕量的定义是
`(d - g·h) / d`，其中 `d` = 簇重心到底面最近边的水平距离，`h` = 重心相对底面的高度。
样例里的 box-D 是 300×200，最窄底面半宽 `d = 100mm`，纵向加速度 `g = 0.8`：

    裕量 ≥ 0  ⟺  100 ≥ 0.8 · h  ⟺  重心高 ≤ 125mm  ⟺  只能码一层

也就是说，要让这类单列堆垛的裕量转正，**只能完全不堆叠**。实测的 -7.8 正是
「2200mm 高的 box-D 单列」代入该式的结果。这也正是 `restraint_mode` 存在的意义：
样例容器是 `unverified`（无固定能力申报），负裕量按设计只告警，真正可执行的输出是
报出来的**所需固定力** `required_stack_*_restraint_kn`——它假定货物有绑扎。

试过的两条路，都已回退：

1. **搭接（错缝码放）让相邻堆垛并成宽底面大簇。** 加了增量并查集簇跟踪器
   （与 `analyze_stack_clusters` 同口径）+ `find_placement` 的「同档择优」钩子：
   在评分元组前缀 `(z, -底面积)` 相同的候选里，按落位后所属簇的安全项择优。
   结果：搭接候选确实存在（10624 次预览中有 275 次能并 2~3 个簇），但最终指标
   **一动不动**（-7.8 / 27 / 11.0），而容器数从 2 涨到 3。
2. **改为最小化所需固定力的边际增量。** 总固定力需求从 11.67 kN 降到 10.05 kN（-14%），
   但**仍然多用 1 个容器**。

两次都多用容器，暴露了一个错误假设：「同 z、同底面积的位置在装填密度上等价，可以自由
拿来换安全性」。**不成立。** 贪心极点法的密度高度依赖角落填充的 `(y, x)` 顺序，一旦
偏离就产生碎片。同档择优这个机制本身就不可用。

第 3 步据此重新定义（见「当前实施状态」）：目标不是裕量转正，而是在**不牺牲容器数**的
前提下降低所需固定力与危险簇数量；且必须先找到一个真正密度中性的调节维度——
候选是放置**顺序**（`order_placeables`）与朝向优先级，而不是同档位置替换。

## 第 5 步：策略质量退化门禁

原先的门禁只挡「装不完 / 有工业错误码 / 容器数 > 3 / 托盘悬空 / 布局不确定」——**只要还
装得完，指标全面变差也照样通过**。阶段四里多用一个容器那次回归，就是靠人工比对基准
才发现的，门禁全程沉默。

现在给每个基准用例的关键指标钉上基线与允许退化幅度（`scripts/benchmark_solver.py`
的 `QUALITY_BASELINES` / `QUALITY_TOLERANCES`），越界即 `RuntimeError` 并指名越界项：

| 指标 | 方向 | 允许退化 |
| --- | --- | --- |
| `container_count` | 越小越好 | 0（容器数是钱，一个都不许多） |
| `total_cost` | 越小越好 | +2% |
| `volume_utilization` | 越大越好 | −2% |
| `stability_score` / `loading_score` | 越大越好 | −3% |
| `required_restraint_kn`（纵向 + 横向） | 越小越好 | +10% |
| `risky_stack_cluster_count` | 越小越好 | +10%，且至少容 1 个 |
| `max_stack_cluster_slenderness` | 越小越好 | +5% |

耗时不入硬门禁：它随机器波动，由 `docs/performance-optimization.md` 单独跟踪。

基线覆盖全部 9 个用例（48 件 ×4、1140 件 ×4，加 `safe_loading` 的安全优先路径）。
`test_every_benchmark_case_has_a_quality_baseline` 保证新增用例不会漏配基线，也保证基线
不会引用没有容差定义的指标（那样会被静默跳过）。

改动确实带来更优解时更新基线——但必须是有意识地改。用两次真实的历史回归验证过门禁确实
能拦下：

- 把 `safe_loading` 的安全优先排序换回纯几何序 → 拦下，报出 `risky_stack_cluster_count`
  27 > 19.7、`max_stack_cluster_slenderness` 11 > 8.4、`required_restraint_kn` 11.67 > 6.08
  等 5 项。
- 把顺序配送评分器退回「横向居中」（即第 4 步修掉的那个缺陷）→ 拦下，报出
  `container_count` 3 > 2、`total_cost` 6000 > 4080、`volume_utilization` 0.346 < 0.509 等 5 项。

## 第 6 步：诊断分层与可执行性语义

原先所有诊断都是 `warning` 一锅端，界面只分「错误 / 提示」两档，用户无法区分
「装不完」「装完但不可执行」「装完可执行但要先绑扎」。而且多只容器共用同一个类型 id，
告警只带 `container_id`，**定位不到具体是哪一只箱子**——去重键又不含实例信息，两只箱子
给出文案相同的告警时，其中一只会被静默吞掉（实测 `TIPPING_RISK` 就丢了一条）。

### 三层语义

`ConstraintViolation.severity` 扩为 `error` / `warning` / `info`，与 `status` 一一对应：

| 层级 | 含义 | 例子 |
| --- | --- | --- |
| `error` | 方案不可执行，或工业模式必填输入缺失。必须解决 | `COG_OUT_OF_RANGE`、`FLOOR_LOAD_EXCEEDED`、`ITEM_WEIGHT_REQUIRED` |
| `warning` | 布局存在物理风险，要绑扎/支挡才能上路 | `TIPPING_RISK`、`STACK_CLUSTER_TIPPING_RISK`、`SECURING_CAPACITY_REQUIRED` |
| `info` | 配置口径与兼容性提示，不要求改动布局 | `STACK_RESTRAINT_UNVERIFIED`、`ISO_COG_TEMPLATE_APPLIED`、`OBJECTIVE_DEPRECATED`、`COST_DATA_MISSING` |

`status` 据此收敛（`_resolve_status`）：

- `infeasible`：存在 `error`；
- `partial`：无 `error` 但有余货——装上的部分可执行，只是没装完；
- `feasible`：无 `error` 且全部装完——可执行；仍可能带 `warning`（需绑扎）或 `info`。

关键是 `feasible` **不等于**「没有风险」。新增 `Solution.diagnostics`
（`error_count` / `warning_count` / `info_count` / `unpacked_count` / `status_reason`）
把这层区别显式化，前端状态标签也随之分成「可执行」与「可执行 · N 项风险」两种。

### 定位到具体设备

`ConstraintViolation` 新增 `container_index`（`solution.containers` 的下标），
由 `finalize_solution` 在每只容器的诊断产出后统一打上。去重键同步纳入
`container_index`，否则两只箱子的同名告警会互相吞掉。前端按 `容器 #n · 货品 x · 站点 s`
展示定位。

### 界面

`ResultPanel` 的诊断区按三层分组展示，每层带一句「该拿它怎么办」的说明；
`EditableTable` 的 AntD `destroyOnClose` 迁移为 `destroyOnHidden`（antd 5.25+），
前端构建无弃用告警。

## 多容器类型混装：按「装完总共多少钱」选箱

先更正一处旧记载：多容器**类型选择**其实早就存在——`run_container_loop` 对
`cost_efficiency` / `space_utilization` 会逐个候选类型试装、择优开箱。此前文档里
「未做类型组合搜索」的说法是错的。

真正的缺口在**选择键**上。原键是 `本箱装载件数 / 本箱成本`，也就是「这一箱每元装几件」。
这是近视的：小箱单价装载效率更高、却装不完，于是被选中，反倒逼出多余的一只箱。
实测（1938 件、20GP 2000 元 / 40GP 3400 元）：

- 原逻辑：40GP + 20GP + 20GP = **7400**
- 穷举最优：40GP + 40GP = **6800**

关键在于**装不完是要额外花钱的**——余货非空时至少还得再开一只箱，而原键完全看不到这一点。
新键改为「本箱成本 + 余货的完成成本下界」（`_score_cost_trials`）：

    估算总价 = 本箱成本 + (余货为空 ? 0 : max(最便宜的一只箱, 余货体积 × 最优单位体积成本))

余货的折算价必须用**各类型试装中实测到的装载体积**来算，不能用容器标称体积：后者等于
假设 100% 填充，会把余货代价低估一半以上（这批货实际只填到 ~51%），结果又会选回那只
装不完的小箱——第一版就是这么错的。修正后 1938 件降到 6800，与穷举最优一致。

### 默认样例新增 40GP

默认样例原本只有 20GP，无从验证类型选择。现新增第二种设备类型：

| 类型 | 内尺寸 (mm) | 体积 | 地板 | 载重 | 启用成本 | 单位体积 |
| --- | --- | ---: | ---: | ---: | ---: | ---: |
| 20GP `cntr` | 5900 × 2350 × 2390 | 33.1 m³ | 13.9 m² | 28000 kg | 2000 | 60.4 /m³ |
| 40GP `cntr40` | 12030 × 2350 × 2390 | 67.6 m³ | 28.3 m² | 26700 kg | 3400 | 50.3 /m³ |

1140 件默认样例上，成本与空间策略改用 **1 只 40GP 取代 2 只 20GP**：

| 指标 | 2 × 20GP | 1 × 40GP |
| --- | ---: | ---: |
| 容器数 | 2 | **1** |
| 总成本 | 4000 | **3400**（−15%） |
| 体积利用 | 51.97% | 50.97% |
| 稳定性 | 43.21% | 44.13% |
| 细长比 | 10.5 | 9.0 |
| 危险堆垛簇 | 26 | 30 |
| 所需固定力 | 14.46 kN | 14.70 kN |
| 耗时 | 6.3 s | 10.3 s |

代价有两处，都已如实入账：货集中到一只箱里堆得更高，危险簇 26 → 30、固定力略升；
耗时翻倍是类型选择的固有开销（每次开箱要对每个候选类型各试装一遍）。安全与配送策略
不做类型选择（它们的目标不是成本），仍用 2 只 20GP，基线未变。

## 通用堆垛簇稳定性

堆垛簇稳定性属于策略无关的物理安全指标。`cost_efficiency`、`space_utilization`、
`safe_loading` 和 `delivery_sequence` 使用完全相同的计算；策略权重不能抵消风险簇。

分析器根据货物和托盘底座之间的垂直接触建立支撑图。每个连通分量作为一个堆垛簇，计算：

- 簇总质量与联合三轴重心；
- 最底层支撑货物形成的支撑边界；
- 纵向和横向运输加速度下的最小整体倾覆裕量；
- 簇高度相对最小支撑宽度的细长比；
- 风险簇数量和最差簇裕量。

输出字段为 `stack_cluster_count`、`risky_stack_cluster_count`、
`stack_cluster_tip_margin` 和 `max_stack_cluster_slenderness`。风险簇生成
`STACK_CLUSTER_TIPPING_RISK`；顺序配送还会在每个站点卸货后重新分析剩余簇，并生成
`POST_DROP_STACK_CLUSTER_TIPPING_RISK`。结果面板显示当前容器的风险簇数量和最差倾覆裕量。

当前指标按自由站立、未额外固定的堆垛计算。容器壁、挡杆、充气袋、缠绕膜和绑扎不能在
没有明确位置与额定能力数据时自动算作支撑。因此负裕量目前作为全策略通用风险警告，而不
直接把方案改成 `infeasible`。后续加入可用固定能力后，工业模式应将“负裕量且固定能力不足”
升级为硬约束。

48件基准首次识别：成本/空间策略各1个风险簇，安全策略2个，顺序配送12个；顺序配送最差
裕量为-5.0、最大细长比7.5。1140件基准识别：成本/空间各26个风险簇，安全策略27个，
顺序配送53个；最差裕量分别约-7.4、-7.8和-8.2。该结果证明旧安全评分没有覆盖截图所示
的高而窄货墙，下一阶段必须把簇指标前置到安全策略候选排序与工业固定能力校验。

### 固定能力闭环

容器新增统一的 `restraint_mode`：

- `unverified`：没有可信的固定装置能力数据，保留方案并告警；
- `none`：明确没有挡杆、绑扎等有效固定能力，按 0 kN 校验；
- `rated`：按纵向、横向额定固定能力分别校验，两个方向都必须满足。

分析器按每个风险簇的质量、联合重心高度、底部支撑边界和运输加速度计算所需抗倾覆力，
并汇总为 `required_stack_longitudinal_restraint_kn` 与
`required_stack_transverse_restraint_kn`。工业候选放置、最终方案以及顺序配送逐站卸货后的
剩余载荷均使用同一能力比较规则。

该约束层对四种生产策略完全通用：`unverified` 生成
`STACK_RESTRAINT_UNVERIFIED`/`STACK_CLUSTER_TIPPING_RISK` 告警；明确为 `none` 或 `rated`
且能力不足时，候选被拒绝并返回 `STACK_CLUSTER_RESTRAINT_INSUFFICIENT`。额定能力足够时风险簇
可以通过，因为物理风险已由已核验的外部固定能力覆盖。前端容器编辑器可配置固定模式和两个
方向的 kN 能力，结果面板同时显示方案实际需要的两个方向能力。

当前能力是“容器级可用总能力”的保守简化模型，尚未描述锚点坐标、绑带角度、挡杆位置、
同一装置能否被多个簇共享以及装置本身安全系数。用于正式工艺放行前，仍需将设备额定值与
企业绑扎标准或工程计算书对应；未完成对应时应保持 `unverified`，不能填写推测能力。
